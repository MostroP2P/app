# Repository Guidelines

## Language

- **Everything published to this repository must be written in English.** This includes code, code comments, identifiers, commit messages, branch names, PR titles and descriptions, review comments, issues, and documentation.
- The **only** exception is user-facing UI copy, which is localized through Flutter l10n ARB files (`lib/l10n/app_{en,es,fr,de,it}.arb`); the source language there is still English (`app_en.arb`).
- Keep English as the default even when discussing changes informally — the public history of the project stays in one language.

## Project Structure & Module Organization

This is **Mostro v2**, a client for the Mostro P2P Bitcoin/Lightning exchange protocol over Nostr. It is a hybrid app: protocol, cryptography, and business logic live in **Rust**, while the **UI is Flutter**, bridged by `flutter_rust_bridge` (FRB).

- `rust/` — Rust core. Nostr protocol, cryptography, keys, relays, persistence, and business logic.
  - `rust/src/api/` — the FRB bridge surface (the only Rust that Dart calls directly). FRB scans **only** `crate::api`.
  - `rust/src/` submodules — `nostr/`, `crypto/`, `mostro/`, `db/`, `nwc/`, `queue/`. Changes here need no codegen unless the `api/` surface changes.
- `lib/` — Flutter shell. UI, navigation, UI state, and device/OS I/O, grouped by feature under `lib/features/<feature>/`.
  - `lib/l10n/` — localization ARB files and generated `AppLocalizations`.
  - `lib/src/rust/` — **generated by `flutter_rust_bridge` — never edit by hand.**
- `test/` — Dart tests, mirroring the feature layout.
- `specs/` — feature specs (`004` = active, `005` = transport v2). `.specify/` holds architecture and constitution docs.
- Platform wrappers live in `android/`, `ios/`, `linux/`, `macos/`, `web/`, `windows/`. Additional docs are in `docs/`.

### The golden rule (architecture)

- **Rust** owns Nostr protocol, cryptography, keys, relays, and business logic.
- **Dart** owns UI, navigation, UI state, and device/OS I/O.
- **No cryptography in Dart.** When in doubt: logic → Rust, device I/O → Dart.

## Build, Test, and Development Commands

Rust is a **library** — there is no `cargo run`; the app is launched through Flutter.

- `cd rust && cargo build` — build the Rust core.
- `cd rust && cargo test && cargo clippy` — mandated verify for Rust (keep clippy warning-free).
- `./scripts/frb-generate.sh` — regenerate the FRB bindings **after any change to `rust/src/api/`**. It refuses to run when your local codegen CLI does not match the version pinned in `pubspec.yaml`.
- `flutter pub get` — sync Dart dependencies after pulling or switching branches.
- `flutter analyze && flutter test` — enforce lints and run the Dart suites; keep the tree warning-free.
- `flutter gen-l10n` — regenerate `AppLocalizations` after editing `lib/l10n/*.arb`.
- `flutter run -d <device>` — launch the app locally on the given device (e.g. `linux`, `chrome`, or `android`).

## Coding Style & Naming Conventions

- **Rust:** follow `rustfmt` defaults and resolve every `clippy` warning. Rust does **not** translate strings — it returns data or a stable marker/code that Dart maps to a localized string.
- **Dart:** follow the Dart formatter defaults (two-space indentation, trailing commas) and resolve every analyzer warning.
- Name Riverpod providers `<Feature>Provider` or `<Feature>Notifier`, and keep feature assets inside their feature directory.
- **Localize all user-facing strings** via ARB files and access them with `AppLocalizations.of(context)` rather than hard-coded literals. Do not hardcode user-facing prose in Rust.
- Wire status strings are **kebab-case** (e.g. `waiting-buyer-invoice`, `fiat-sent`).

## Testing Guidelines

- Place Dart tests under `test/`, mirroring the feature layout, using the `*_test.dart` suffix.
- Place Rust tests alongside the code they cover; run `cargo test` before pushing.
- Run `flutter analyze && flutter test` and `cd rust && cargo test && cargo clippy` before opening a PR.
- Add targeted scenarios when expanding complex logic, async workflows, or protocol handling.

## Commit & Pull Request Guidelines

- Use **conventional commits** (`feat`, `fix`, `docs`, `refactor`, `chore`, `test`, etc.) with an optional scope, e.g. `feat(orders): …`. Write concise, imperative subjects.
- Branch names follow `type/kebab-desc` (e.g. `docs/add-agents-contributing`).
- **One PR per feature.** Long features are split into phased PRs rather than a single big-bang change. Everything lands via **pull request to `main`**.
- PR descriptions should capture intent, list key changes, link tracking issues, and flag risk areas; include command output for manual tests and screenshots for UI updates.
- Keep branches rebased and free of merge commits. PRs are reviewed with CodeRabbit.
- See [CONTRIBUTING.md](CONTRIBUTING.md) for the full contributor workflow and review conventions.

## Documentation & Markdown

- Add a language specifier to every fenced code block (markdownlint MD040 flags blocks without one). Example:

  ```sh
  flutter test
  ```

- **Generated code is not documentation to edit:** never hand-edit `lib/src/rust/`. Regenerate it with `./scripts/frb-generate.sh`.
- Specs are a **living artifact** — update the matching spec/contract in `specs/` or `.specify/` as part of any behavior or contract change.
